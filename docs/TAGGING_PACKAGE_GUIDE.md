# Tagging Architecture Guide

## 목적

이 문서는 SOI의 태깅 구조를 쉬운 말로 설명하는 문서입니다.

예전에는 태깅 로직 안에 SOI 전용 댓글 규칙이 많이 들어 있었습니다.
지금은 이 구조를 정리해서, 다른 서비스나 다른 플랫폼에서도 태깅 엔진을 가져다 쓸 수 있게 만들었습니다.

먼저 가장 중요한 점부터 적으면:

- `tagging_core`는 공용 태그 엔진
- `tagging_flutter`는 Flutter용 드래그/오버레이 표면
- `tagging_soi`는 SOI 전용 연결 모듈

입니다.

여기서 주의할 점은:

- `tagging_core`와 `tagging_flutter`는 실제 `packages/...` 아래의 패키지입니다.
- `tagging_soi`는 패키지가 아닙니다.
- `tagging_soi`는 `lib/features/tagging_soi` 아래에 있는 앱 내부 feature/module입니다.

---

## 가장 쉬운 설명

태깅 기능은 크게 4단계로 나뉩니다.

1. 사용자가 태그 내용을 만든다.
2. 사용자가 그 태그를 포스트 위로 드래그해서 위치를 정한다.
3. 서버에 저장한다.
4. 저장된 태그를 다시 화면에 보여준다.

이 4단계 중에서,

- 공통으로 쓸 수 있는 부분은 `tagging_core`
- Flutter에서 보여주는 부분은 `tagging_flutter`
- SOI만 아는 댓글 규칙은 `lib/features/tagging_soi`

가 맡도록 나눴습니다.

---

## 구성요소별 역할

### 1. `tagging_core`

여기는 진짜 핵심 엔진입니다.

이 패키지는 이런 것만 압니다.

- 지금 작성 중인 태그 draft가 무엇인지
- 태그가 포스트의 어디에 붙을지
- 서버로 어떤 저장 요청을 보낼지
- 저장이 끝난 뒤 overlay/thread 캐시를 어떻게 갱신할지

반대로 이 패키지는 이런 것은 모릅니다.

- SOI 댓글 모델
- 프로필 이미지 URL 규칙
- SOI 이모지 정책
- SOI 답글 규칙
- SOI의 API 호출 방법

즉, `tagging_core`는 "태그 엔진"만 남긴 상태입니다.

---

### 2. `tagging_flutter`

여기는 Flutter에서 태그를 보여주고 움직이게 하는 데 필요한 공용 위젯만 둡니다.

지금 남아 있는 것은 이런 것들입니다.

- 태그 오버레이 표시
- 드래그 핸들
- 좌표/버블 배치 계산

반대로 이런 것은 여기서 뺐습니다.

- SOI 스타일 입력창
- SOI용 기본 바
- SOI용 댓글 컴포저

이유는 간단합니다.
입력 UI는 플랫폼마다 거의 다르게 생기기 때문입니다.

---

### 3. `tagging_soi`

여기는 패키지가 아니라 SOI 앱 안의 전용 모듈입니다.

실제 위치는 아래입니다.

- `lib/features/tagging_soi`

쉽게 말하면 `tagging_core`와 SOI 앱 사이의 통역사입니다.

이 모듈은 이런 일을 합니다.

- SOI의 `Comment`를 `TagEntry`로 바꿉니다
- 저장 요청을 SOI 댓글 API 호출로 바꿉니다
- SOI의 reply, profile, media, emoji 같은 규칙을 처리합니다
- SOI 화면에서 쓰는 입력 UI를 제공합니다

즉, SOI만 아는 정보는 다 여기서 처리합니다.

---

## 왜 이렇게 나눴는가

이 구조의 목표는 아주 단순합니다.

`tagging_core`를 다른 서비스에서도 그대로 쓸 수 있게 만들기 위해서입니다.

예를 들어 어떤 다른 서비스가 있다고 가정해봅시다.

- 댓글이 아니라 메모를 태그할 수도 있고
- 이미지가 아니라 텍스트 카드에 태그할 수도 있고
- 프로필 이미지가 없을 수도 있고
- 이모지가 아예 없을 수도 있습니다

이런 서비스가 SOI용 댓글 모델까지 같이 끌고 가면 너무 무겁고 어색합니다.

그래서 공통 부분만 `tagging_core`에 남기고,
서비스 전용 부분은 adapter로 분리했습니다.

---

## 지금 구조를 비유하면

예전 구조는 이런 느낌이었습니다.

- 엔진 안에 SOI 전용 핸들, 좌석, 내비게이션이 같이 들어 있는 차

지금 구조는 이런 느낌입니다.

- 엔진은 엔진만 따로 분리
- SOI용 부품은 SOI 쪽에 따로 보관

그래서 다른 차를 만들 때는 엔진만 가져다가 새 부품을 붙이면 됩니다.

---

## 실제 유저 플로우와 책임 분리

사용자 흐름은 보통 이렇게 갑니다.

1. 사용자가 입력 UI에서 태그 내용을 만든다
2. 사용자가 그 태그를 포스트 위로 드래그한다
3. 포스트 위에 드롭해서 위치를 정한다
4. 서버에 저장한다
5. 저장된 태그를 다시 포스트 위에 보여준다

각 단계의 담당은 이렇게 나뉩니다.

### 입력

- SOI에서는 `lib/features/tagging_soi`의 UI가 담당
- 다른 플랫폼에서는 그 플랫폼 UI가 직접 담당

### 드래그 위치 계산

- `tagging_core`
- `tagging_flutter`

### 저장 요청 만들기

- `tagging_core`

### 실제 서버 저장

- `lib/features/tagging_soi`
- 또는 다른 서비스의 adapter

### 저장 후 다시 보여주기

- `tagging_core`가 캐시 갱신 흐름 담당
- `tagging_flutter` 또는 플랫폼 UI가 렌더링 담당

---

## 새 플랫폼이 태깅 기능을 붙일 때 필요한 것

다른 플랫폼에서 태깅을 붙이고 싶다면, 크게 3가지만 있으면 됩니다.

### 1. 입력 UI

그 플랫폼에 맞는 입력 화면을 만들면 됩니다.

예를 들면:

- 텍스트만 입력
- 텍스트 + 이미지
- 텍스트 + 영상
- 음성까지 포함

이건 서비스마다 다르기 때문에 공용 패키지에 넣지 않습니다.

### 2. 저장 어댑터

그 플랫폼 서버에 맞는 adapter를 하나 만들면 됩니다.

즉,

- `TagSaveRequest`를 받아서
- 그 서비스 API로 저장하고
- 저장 결과를 `TagEntry`로 돌려주는 코드

만 있으면 됩니다.

### 3. 조회 어댑터

서버에서 저장된 태그를 다시 가져와서 `TagEntry`로 바꿔주는 adapter가 필요합니다.

즉,

- 서버 응답 모델 -> `TagEntry`
- `TagEntry` -> 화면 표시

흐름만 연결해주면 됩니다.

---

## core에 넣지 않기로 한 것

다음 것들은 공용 엔진에 넣지 않는 것이 맞습니다.

- SOI 댓글 타입
- 프로필 이미지 presigned URL 처리
- SOI 이모지 선택 상태
- SOI reply 규칙
- 서비스별 API 모델
- 서비스별 입력 UI

이런 것까지 core가 알기 시작하면, 다시 SOI 전용 로직이 core 안으로 들어오게 됩니다.

---

## SOI에서 지금 남아 있는 역할

SOI는 여전히 자기만의 규칙을 가집니다.

예를 들면:

- `Comment` 모델
- reply/parent 관계
- 프로필 이미지 표시 방식
- media key/url 처리
- 이모지 상태
- SOI 스타일 댓글 입력 UI

이런 것들은 없어지지 않았습니다.
다만 위치가 `tagging_core` 안이 아니라 `tagging_soi`로 이동한 것입니다.
정확히는 `lib/features/tagging_soi`로 이동한 것입니다.

즉, 기능을 지운 것이 아니라 책임 위치를 옮긴 것입니다.

---

## 지금 구조의 장점

### 1. 재사용하기 쉬워졌습니다

새 플랫폼은 SOI 댓글 구조를 몰라도 됩니다.
그냥 `TagEntry`, `TagSaveRequest` 계약만 맞추면 됩니다.

### 2. core가 더 작고 명확해졌습니다

core는 "태그 엔진"이라는 자기 역할에만 집중합니다.

### 3. 서비스별 정책이 분리됐습니다

SOI 규칙이 바뀌어도 `tagging_soi`를 주로 보면 됩니다.
다른 서비스는 자기 adapter만 관리하면 됩니다.

### 4. 테스트하기 쉬워졌습니다

공통 엔진 테스트와 SOI adapter 테스트를 분리해서 볼 수 있습니다.

---

## 기억하면 좋은 문장

### `tagging_core`

"태그를 어떻게 들고 있고, 어디에 붙이고, 어떻게 저장 흐름을 만들지"를 담당한다.

### `tagging_flutter`

"Flutter에서 태그를 어떻게 보여주고 드래그할지"를 담당한다.

### `tagging_soi`

"SOI 댓글 시스템과 core를 어떻게 연결할지"를 담당한다.

---

## 결론

이번 리팩터링의 핵심은 기능을 나눈 것이 아니라 책임을 제대로 나눈 것입니다.

이제 구조는 이렇게 이해하면 됩니다.

- `tagging_core`: 공용 태그 엔진 패키지
- `tagging_flutter`: Flutter용 공용 surface 패키지
- `lib/features/tagging_soi`: SOI 전용 adapter와 UI 모듈

그래서 앞으로 다른 플랫폼에 태깅을 붙일 때는,

- `tagging_core`는 그대로 재사용하고
- 그 플랫폼용 입력 UI와 adapter만 새로 만들면 됩니다.

이게 이번 구조 정리의 가장 큰 의미입니다.
